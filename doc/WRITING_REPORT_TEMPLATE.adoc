== Overview

VulnScout uses https://jinja.palletsprojects.com/[Jinja2] as its templating engine.
Templates can produce any text-based format: AsciiDoc, HTML, Markdown, CSV, plain text, etc.

=== Template Locations

[cols="1,2"]
|===
|Path |Purpose

|`src/views/templates/`
|Built-in templates shipped with VulnScout.

|`/scan/templates/`
|Custom templates inside the container (staged automatically when a path is passed to `--report`).

|`.vulnscout/templates/`
|Per-project host-side templates (mounted automatically if the directory exists).
|===

=== Adding a Custom Template

Pass the file path directly to `report` — it stages and runs the template in one step:

[source,shell]
----
./vulnscout --project demo report /path/to/my-report.adoc
----

=== Automatic Generation

Set the `GENERATE_DOCUMENTS` environment variable to automatically build templates at the end of every scan.
Multiple templates are comma-separated (values are trimmed):

[source,shell]
----
./vulnscout config GENERATE_DOCUMENTS "summary.adoc, all_assessments.adoc"
----

Generated files are written to the outputs directory (default: `.vulnscout/outputs/`).
All templates can also be run on-demand from the web interface using the export button in the toolbar.

== Global Variables

The following variables are available in every template:

[cols="1,1,2"]
|===
|Variable |Type |Description

|`vulnerabilities`
|dict
|All vulnerabilities found in the project, keyed by ID. Use `\| as_list` to get a list.

|`packages`
|dict
|All packages found in the project, keyed by ID. Use `\| as_list` to get a list.

|`assessments`
|dict
|All assessments found in the project, keyed by ID. Use `\| as_list` to get a list.

|`author`
|string
|Company name producing this document (from `AUTHOR_NAME` config).

|`export_date`
|string
|Date of export as `YYYY-MM-DD`.

|`scan_date`
|string
|Date and time of the last scan, or `unknown date` if unavailable.

|`client_name`
|string
|Customer company name. Set via `CLIENT_NAME` config or overridden per-export from the web UI. May be empty.

|`unfiltered_vulnerabilities`
|dict
|All vulnerabilities, bypassing any active export filter. Use `\| as_list` to get a list. Always the full dataset.

|`unfiltered_assessments`
|dict
|All assessments, bypassing any active export filter. Keyed by assessment ID.

|`failed_vulns`
|list[string]
|List of vulnerability IDs that triggered the `--match-condition` expression. Empty list when no condition was set or no vulnerability matched. Use `unfiltered_vulnerabilities[vuln_id]` to get the full object.
|===

=== Environment Variables in Templates

Use `env(name, default="")` to read host environment variables inside a template:

[source,jinja2]
----
{{ env("DISTRO") }}
{{ env("DISTRO", "unknown") }}
----

VulnScout automatically scans templates for `env("...")` patterns and passes the matching variables from the host.

==== Status Values

[cols="1,2"]
|===
|Status |Meaning

|`in_triage`, `under_investigation`
|Found but not yet confirmed. Default status. Matched by `status_pending` / `status_active`.

|`affected`, `exploitable`
|Confirmed and affecting the product. Matched by `status_affected` / `status_active`.

|`fixed`, `resolved`, `resolved_with_pedigree`
|Fixed and no longer exploitable. Matched by `status_fixed` / `status_inactive`.

|`not_affected`, `false_positive`
|False positive or does not affect the product. Matched by `status_ignored` / `status_inactive`.
|===

==== Justification Values

`component_not_present`, `vulnerable_code_not_present`, `vulnerable_code_not_in_execute_path`,
`vulnerable_code_cannot_be_controlled_by_adversary`, `inline_mitigations_already_exist`,
`code_not_present`, `code_not_reachable`, `requires_configuration`, `requires_dependency`,
`requires_environment`, `protected_by_compiler`, `protected_at_runtime`,
`protected_at_perimeter`, `protected_by_mitigating_control`.

==== Response Values

`can_not_fix`, `will_not_fix`, `update`, `rollback`, `workaround_available`.

== Filters and Helpers

In addition to https://jinja.palletsprojects.com/en/3.1.x/templates/#list-of-builtin-filters[Jinja2 built-in filters], the following custom filters are available.

=== Conversion

[cols="1,2"]
|===
|Filter |Description

|`as_list`
|Convert a dict to a list using `.values()`.

|`print_iso8601`
|Convert an ISO 8601 duration string to a human-readable form (e.g. `P2DT4H` → `2d 4h`) or format a datetime string.

|`limit(n)`
|Return at most `n` results from a list.
|===

=== Status Filtering

[cols="1,2"]
|===
|Filter |Description

|`status(x)`
|Keep vulnerabilities whose status is in `x` (string or list of strings).

|`status_pending`
|Shorthand for `in_triage` + `under_investigation`.

|`status_affected`
|Shorthand for `affected` + `exploitable`.

|`status_fixed`
|Shorthand for `fixed` + `resolved` + `resolved_with_pedigree`.

|`status_ignored`
|Shorthand for `not_affected` + `false_positive`.

|`status_active`
|`status_pending` + `status_affected`.

|`status_inactive`
|`status_fixed` + `status_ignored`.
|===

=== Score Filtering

[cols="1,2"]
|===
|Filter |Description

|`severity(x)`
|Keep vulnerabilities with severity in `x` (string or list of strings).

|`epss_score(x)`
|Keep vulnerabilities with EPSS score ≥ `x`. `x` is a percentage in [0, 100].
|===

=== Date Filtering

Both `last_assessment_date` and `filter_by_publish_date` accept date expressions in the following formats:

* `'>2026-01-01'` — after this date (exclusive)
* `'>=2026-01-01'` — on or after this date (inclusive)
* `'<2026-01-01'` — before this date (exclusive)
* `'<=2026-01-01'` — on or before this date (inclusive)
* `'2026-01-01..2026-01-31'` — between two dates (both inclusive)
* `'2026-01-01'` — exact date match

[cols="1,2"]
|===
|Filter |Description

|`last_assessment_date(x)`
|Filter vulnerabilities by their last assessment date.

|`filter_by_publish_date(x, include_unknown=False)`
|Filter by CVE publish date. Pass `include_unknown=True` to also include vulnerabilities with no known publish date.
|===

=== Sorting

[cols="1,2"]
|===
|Filter |Description

|`sort_by_epss`
|Sort by EPSS score, highest first.

|`sort_by_effort`
|Sort by `effort.likely`, most effort first.

|`sort_by_last_modified`
|Sort by most recent assessment date, newest first.
|===

== Examples

=== Count active vulnerabilities

[source,jinja2]
----
{{ vulnerabilities | as_list | status_active | length }}
----

=== Highest EPSS active vulnerability

[source,jinja2]
----
{{ vulnerabilities | as_list | status_active | sort_by_epss | first }}
----

=== Vulnerabilities assessed after a date

[source,jinja2]
----
{{ vulnerabilities | as_list | last_assessment_date('>2026-01-01') | length }}
----

=== All critical unresolved vulnerabilities, sorted by most recent assessment

[source,jinja2]
----
{% for vuln in vulnerabilities | as_list | severity('critical') | status_active | sort_by_last_modified %}
- {{ vuln.id }} (CVSS: {{ vuln.severity.max_score }})
{% endfor %}
----

=== Use an environment variable

[source,jinja2]
----
Product: {{ env("PRODUCT_NAME", "unknown") }}
Version: {{ env("PRODUCT_VERSION", "unknown") }}
----
